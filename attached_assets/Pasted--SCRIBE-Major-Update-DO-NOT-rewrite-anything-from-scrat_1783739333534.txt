## PHASE 1 — Fix: Cursor & Screen Jumping in Editor

```
PHASE 1: Fix cursor and scroll jumping in the editor

This is the most disruptive bug in the app. The screen or cursor jumps unexpectedly 
in two situations:
  A. After pressing Enter inside brackets or quotation marks (the shortcut bar 
     auto-closes them and moves the cursor out), the view scrolls to an unexpected 
     position.
  B. During normal typing and when adding line breaks, the visible area jumps up 
     or down randomly.

Root causes to investigate and fix:

1. KEYBOARD AVOIDING JUMP:
   Check how the editor's ScrollView or KeyboardAvoidingView is configured.
   - If using KeyboardAvoidingView with behavior="padding" or behavior="height", 
     this can cause the view to resize and jump when the keyboard appears/shifts.
   - Replace with a layout that uses the "软键盘" pattern: listen to 
     Keyboard.addListener('keyboardDidShow') and manually set a paddingBottom on 
     the scroll container equal to the keyboard height. This prevents the layout 
     from resizing and jumping.
   - Use KeyboardAvoidingView behavior="padding" ONLY on iOS. On Android, set 
     behavior={undefined} and handle it manually via Keyboard listeners.

2. SCROLL POSITION JUMP ON CURSOR MOVE:
   - When the shortcut bar triggers a cursor position change (bracket close + Enter), 
     the ScrollView may auto-scroll aggressively.
   - After any programmatic cursor/selection change via Selection state, call 
     scrollView.scrollTo() with animated: false to prevent the jump animation.
   - Add a 50ms setTimeout before any scroll adjustment that follows a shortcut 
     bar action, to let the layout settle first.

3. AUTOSCROLL DURING TYPING:
   - Ensure the TextInput for the editor has scrollEnabled={false} if it is inside 
     a ScrollView. The ScrollView should handle all scrolling, not the TextInput itself.
   - Having both the TextInput and its parent ScrollView trying to scroll simultaneously 
     causes the random jumping.

4. LINE BREAK JUMP:
   - When the user presses Enter near the bottom of the visible area, the view should 
     scroll just enough to keep the new line visible — not jump to the top or bottom.
   - Implement this by tracking cursor Y position using onSelectionChange and only 
     calling scrollView.scrollTo() when the cursor moves outside the currently 
     visible area (below visibleBottom or above visibleTop).

Test all four fixes together. Confirm: typing, Enter key, bracket/quote auto-close, 
and multi-line breaks all behave without any jumping.
```

## PHASE 2 — Fix: Markdown Rendering (Bold, Italic, Headers)

```
PHASE 2: Make markdown formatting work in the editor

Currently markdown syntax is typed but nothing renders — text does not become bold, 
italic, or a header. Fix this.

The approach: use a hybrid editor model.
- The underlying data stored in files stays as plain markdown text (e.g. **bold**).
- The editor DISPLAYS formatted output in real time using a custom renderer.
- This is called a "live preview" or "WYSIWYG-lite" markdown editor.

Implementation:

1. DO NOT replace the TextInput with a WebView or a heavy library. Keep the existing 
   TextInput for input.

2. Add a real-time markdown renderer that runs on the displayed text:
   Use react-native-marked or a lightweight custom parser to convert the markdown 
   content to styled React Native Text components.
   
   If react-native-marked is already in package.json, use it.
   If not, implement a lightweight custom renderer supporting these formats only:
     # Header 1       → large bold text, bigger font size
     ## Header 2      → medium bold text
     ### Header 3     → slightly larger bold text
     **bold**         → bold Text component
     *italic*         → italic Text component
     ***bold italic** → bold + italic Text component
     > blockquote     → left-bordered indented text
     --- or ***       → horizontal rule / scene break line
     Regular text     → normal paragraph

3. RENDERING STRATEGY:
   - Split the content by line breaks.
   - For each line, detect the markdown pattern and wrap it in the correct Text style.
   - Render the result in a read-only Text view that overlays or sits directly below 
     the TextInput.
   
   PREFERRED approach given the existing editor structure:
   - Use a "write mode / read mode" toggle:
     - Write mode: shows the raw TextInput with markdown syntax visible (for editing).
     - Read mode (Preview): shows the beautifully rendered markdown output.
     - Toggle button in the editor toolbar (an eye icon): switches between modes.
     - This is how iA Writer and Ulysses handle it on mobile — clean and reliable.

4. ALTERNATIVELY, if the existing Editor.tsx already attempts inline rendering:
   - Read the current approach first.
   - Fix whatever is broken rather than replacing the whole system.

5. The shortcut bar's ** bold **, * italic * buttons must insert the correct markdown 
   syntax around selected text, or at cursor position if nothing is selected.
   If this is not already working, fix it in this phase.
```

## PHASE 3 — Fix: Connect-a-Folder Showing Folders as TXT Files

```
PHASE 3: Fix the Connect Folder feature — folders appearing as TXT files

The existing app has two storage modes:
  1. Local Vault — files and folders created inside the app's private storage.
  2. Connect Folder — user picks any folder from their phone using a folder picker, 
     and the app reads/writes there. This lets users survive an uninstall by keeping 
     files on phone storage.

The bug: in the Connect Folder file explorer, ALL folders are showing up as TXT files. 
Clicking them open it in edit mode. Actual text files are showing and working correctly.

Fix:

1. READ THE EXISTING FileTree.tsx AND safStorage.ts FULLY before touching anything.
   Understand exactly how the connected folder's contents are being read.

2. THE BUG: When reading directory contents using expo-file-system's 
   FileSystem.readDirectoryAsync(), it returns a flat list of names (strings) — 
   it does NOT tell you which items are files vs folders.
   
   To distinguish files from folders, you must call FileSystem.getInfoAsync(itemPath) 
   on each item and check the isDirectory property in the result.
   
   The current code is likely treating all items as files. Fix it by:
   - For each item name returned by readDirectoryAsync(), call getInfoAsync() on the 
     full path.
   - If result.isDirectory === true → render it as a FOLDER (tappable, expandable, 
     shows folder icon, no file extension shown).
   - If result.isDirectory === false → render it as a FILE (show name, tappable to open).

3. Make these getInfoAsync() calls in parallel using Promise.all() for performance — 
   do NOT await them one by one in a loop.

4. FOLDER BEHAVIOR after fix:
   - Tapping a folder expands it inline (same as Local Vault folders do).
   - Folders show a folder icon matching the existing UI style.
   - Files inside show correctly with their file type icon.
   - Empty folders, that don't have any text/md files shouldn't show.

5. FILTER: Only show .txt and .md files. Silently ignore all other file types 
   (.jpg, .pdf, system files etc.) so the explorer stays clean.

6. The Local Vault explorer must not be touched — only fix the Connect Folder path.
```

## PHASE 5 — Autosave + Version History + Crash Recovery

```
PHASE 5: Autosave, version history, crash recovery

1. AUTOSAVE:
   - Autosave the open file every 30 seconds IF content changed since last save.
   - Use a 2000ms debounce on the editor's onChange — do NOT save on every keystroke.
   - Show a small status indicator in the top bar:

   - History screen: list of saved timestamps, tap one to preview that version.
   - "Restore this version" button with a confirmation dialog before overwriting.

3. CRASH RECOVERY:
   - On every autosave, also write content + file path to:
     [appDocumentDirectory]/.recovery/lastSession.json

   - On app launch, check if .recovery/lastSession.json exists.
   - If yes, show a non-intrusive banner at top: 
     "Unsaved session found — Restore?" with [Restore] and [Dismiss] buttons.
   - Restore: reopen that file and load the recovered content.
   - Dismiss: delete the recovery file silently.
```

```
   - Confirm autosave interval, word count throttle, and keyboard listeners

7. IMPORTS:
   - Find any imports that pull in an entire library when only one function is needed.
```
